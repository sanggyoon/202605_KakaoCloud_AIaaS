# Peakly 부하 테스트 리포트

- 일자: 2026-06-22
- 도구: k6 (단일 부하 VM, KakaoCloud 별도 VPC, `210.109.14.1`)
- 스크립트: `loadtest/peakly-rampup.js` (FE 페이지 + Supabase 조회 + `/api/visit`)
- 시나리오: 램프 50 → 200 → 500 VU, 총 약 16분, p95>3s·에러>5% 자동중단(미발동)
- 대상: `peakly.art`(FE, K3s+HPA), `data.peakly.art`(vm4 Supabase)

## 한 줄 결론

**HPA 자동확장이 의도대로 동작했고(앱 파드는 CPU 70% 타깃에서 안정), 1차는 500 VU까지
자동중단 없이 견뎠다(p95 2.12s·에러 0%). 2차 붕괴점 탐색(읽기 핫패스)에서 ~700 동시
사용자에서 vm4(DB) CPU 100% 포화로 한계 도달 — 즉 시스템 천장은 자동확장되는 앱 티어가
아니라 단일 노드 vm4(DB)다.** 설계 spec의 가설("DB가 최종 천장")이 측정으로 입증됨.

> 측정 요약: 1차(전체경로 ~500 VU) 견딤·vm4 97% 근접 / 2차(읽기 ~713 VU) vm4 100% 붕괴 /
> 3차(캐시 개선 후) 2000 VU 완주·vm4 ~20%·p95 0.66s — DB 천장 제거 /
> **4차(FE max16, 고VU) ~3600 VU 붕괴 — 천장이 app 노드(vm2/vm3 CPU 100% + Pending 5)로 이동.**
> 붕괴점 추이: ~700(DB) → ~3600(app 노드), 약 5배. 상세는 아래 각 섹션 참고.

### 노드 CPU 피크 (부하 13:30~13:44)

| 노드 | 내부 IP | 피크 CPU | 판정 |
|---|---|---|---|
| **vm4 (DB)** | 10.1.5.10 | **~97%** | 거의 포화 — **다음 천장** |
| **vm1 (Ingress)** | 10.1.1.10 | ~85% | 진입점, 두 번째 부하 |

> IP↔노드 매핑은 `kubectl get nodes -o wide`로 확정.

### 천장 판정

- **노드(app) 천장 아님**: Pending이 스케일아웃 시작 시 1개 순간 발생 후 0 유지 → app 노드
  용량은 충분했고 파드 스케줄링 정상.
- **단일 노드(DB·인그레스)가 천장 후보**: app 파드는 HPA가 70%로 눌렀지만, 자동확장되지
  않는 vm4(~97%)·vm1(~85%)이 한계에 근접. 더 밀면 여기가 먼저 막힘.

## 결론

1. **HPA 검증 완료** — FE 2→6, BE 1→4 자동확장 및 자동 복귀까지 정상.
2. **현재 한계** — 500 VU에서 p95 2.12s·에러 0%로 **여전히 건강**. 쾌적 운영점은 대략
   300~400 VU 추정(꼬리 지연 상승 시작 구간).
3. **다음 천장** — app 노드가 아니라 **vm4(DB)·vm1(인그레스) 단일 노드 CPU**.

## 2차 — 붕괴점 탐색 (읽기 핫패스)

1차가 500 VU에서 자동중단 없이 끝나(천장 미도달) 붕괴점을 직접 찾기 위해 추가 실행.

### 시행착오 (배운 점)
- **급경사 램프(`peakly-stress.js`, 1분에 400 VU)** 는 **31초/~210 VU에서 abort** — `/api/visit`
  p95 5s가 원인. BE가 min 1개인 채 급스파이크를 맞아 **HPA가 스케일업하기 전에** 막힌 것.
  서비스의 진짜 한계가 아니라 "BE@1pod는 급증을 못 받는다"는 artifact. → 워크로드/램프 재설계.

### 본 측정 (`peakly-stress-read.js` — visit 제거 + 완만 램프)
실제 사용자 핫패스(FE 페이지 + Supabase movie 읽기)만, 200→2000 VU 완만 램프.

| 지표 | 값 |
|---|---|
| 붕괴 지점 | **~713 VU** 에서 자동중단 (p95 3s 도달) |
| `db_movies_ms` | avg 1.01s · med 239ms · p90 3.01s · **p95 4.53s** · max 10.2s |
| `fe_page_ms` | avg 13ms · **p95 22ms** (FE는 HTML 서빙뿐 — 무부하) |
| 에러율 | 0.22% (DB 231건 — 포화 시점 타임아웃 추정) |
| 처리량 | 318 req/s · 157 iter/s |

### 노드/오토스케일 (Grafana)
- **vm4(DB, 10.1.5) CPU 100% 핀** (피크 구간) ← 명확한 천장
- vm1(ingress) ~70%, app(vm2/vm3) ~40%, **Pending 0**, FE 2→6 정상 확장
- **부하 생성기 결백**: 병목이 생성기였다면 모든 경로가 느려야 하나, `fe_page`는 22ms로
  멀쩡하고 `db_movies`만 4.53s → 느림이 DB 경로에 국한. 수신 27MB/s로 여유 → 서비스 측정 유효.

### 붕괴점 결론
- **읽기 핫패스 한계 ≈ 600~700 동시 사용자**, 한계 원인은 **단일 노드 vm4(DB) CPU 포화**.
- 앱 티어(FE)는 HPA로 잘 확장(2→6, 노드 여유) → **시스템 천장은 앱이 아니라 DB(vm4) 확정.**
- 쾌적 운영점 추정 ~400~500 VU.

## 3차 — 캐시 개선 후 검증 (`/api/movies` 경로)

DB 부하 감소 작업(라우트 캐싱 1h + 페이로드 축소, spec `2026-06-22-db-load-cache-payload`)
적용 후 효과 측정. **주의:** `peakly-stress-read.js`는 vm4(`data.peakly.art`)를 직격해
캐시를 우회하므로 효과가 안 보인다(재실행 시 동일하게 vm4 100% 포화). 실제 브라우저 동작인
**FE `/api/movies`(캐시) 경로**를 치는 `peakly-stress-cached.js`로 측정해야 한다.

### 결과 (`peakly-stress-cached.js`, 200→2000 VU 완만 램프)

| 지표 | 개선 전(읽기 직격) | **개선 후(캐시 경로)** |
|---|---|---|
| 결과 | ~713 VU 자동중단 | **2000 VU 완주, 중단 없음** |
| p95 | 4.53s(db) | **657ms** |
| 처리량 | 318 req/s | **990 req/s** |
| 에러율 | 0.22% | 0.00% (77만 건 중 54건) |
| **vm4(DB) CPU** | **100% 포화** | **~20–25% 평탄** |
| FE 레플리카 | 2→6 | **2→8(max)** |
| Pending | 0 | 0 |

### 노드 거동 (Grafana)
- **vm4(10.1.5)**: 부하 내내 ~20–25% 평탄 → **DB 천장 제거 확인**(캐시 적중으로 DB 미접촉).
- app 노드 vm2(10.1.3)·vm3(10.1.4): ~90–100%로 상승, FE 파드가 캐시·HTML 서빙을 흡수.
- ingress vm1(10.1.1): ~85%. vm5(10.1.7): ~0%.
- FE HPA가 **max 8**까지 확장(이전 6 → 8), Pending 0.

### 결론
- **캐싱으로 단일 DB(vm4) 천장 제거.** 동일 인프라에서 한계가 **~700 → 2000+ 동시 사용자**로
  최소 3배 향상, p95 **4.5s → 0.66s**.
- 새 천장은 **자동확장 가능한 app 티어(FE 파드/노드)** 로 이동 — FE max 상향/노드 증설로 확장 가능.
- (참고) 캐시 미적용 부하 도구(`peakly-stress-read.js`)는 vm4 직격이라 효과 측정에 부적합.
  캐시 경로 측정은 `peakly-stress-cached.js` 사용.

## 4차 — FE maxReplicas 16 상향 후 붕괴점 (캐시 경로, 고VU)

3차(캐시)가 2000 VU에서 안 깨져(붕괴점 미도달), FE HPA `maxReplicas` 8→16으로 올리고
`peakly-stress-cached-high.js`로 1000→5000 VU까지 밀어 현재 붕괴점을 찾음.

### 결과
| 지표 | 값 |
|---|---|
| 붕괴 지점 | **~3600 VU** 에서 자동중단(p95 3.02s) |
| 처리량 | 1196 req/s |
| 에러율 | 0.29% |
| `api_movies_ms` p95 | 2.53s (캐시 경로) |
| `fe_page_ms` p95 | 3.53s (FE HTML 서빙 — 더 느림) |

### 노드/오토스케일 (Grafana)
- **vm4(DB, 10.1.5): ~20–25% 평탄** — 캐시 계속 작동, DB 무관.
- **app 노드 vm2(10.1.3)·vm3(10.1.4): ~100% 포화.**
- FE 레플리카 2→**16(max)** 시도하나 **Pending 5개 지속** → 실제 ~11개만 스케줄.
- ingress vm1 ~85%, vm5 0%.

### 붕괴점 결론
- **현재 천장 = app 노드(vm2/vm3) 용량** — CPU 100% + Pending 5. DB는 더 이상 천장 아님.
- **FE max 추가 상향은 무의미**(16 원했으나 노드가 11개만 수용). 다음 capacity 확장은
  **app 워커 노드 증설(VM 추가/vCPU 상향)** 이 정답.
- 붕괴점 추이: **~700(DB) → 2000 견딤 → ~3600(app 노드)**, 캐시 도입으로 약 5배 향상.

## 다음 개선 (범위 밖 / 후속 과제)

1. **DB(vm4) — 최우선** — 단일 노드 CPU가 전체 천장. 읽기 replica, 커넥션 풀링(PgBouncer),
   **영화목록 응답 캐싱**(`select=*&limit=120` 대용량 → CDN/Edge 캐시 효과 큼), 인덱스·쿼리 튜닝.
2. **BE `/api/visit`** — min 1→2 상향 또는 급증 대비(예열). 단 실제 visit은 방문자당 1회라 우선순위 중.
3. **인그레스(vm1)** — 피크 ~70%, 여유 있으나 DB 개선 후 재측정 시 관찰.
4. (선택) k6 → Prometheus remote-write로 클라이언트 지연을 Grafana에 통합 / 부하 VM 다중화로
   생성기 한계 제거 후 DB 절대 한계 재측정.
