## 2. 네트워크 구성

### 트래픽 흐름

```
[인바운드 HTTPS]
사용자 브라우저
  → DNS (peakly.art, Route53 → 210.109.83.10)
  → vm1 Public IP:443
  → Ingress-Nginx Pod (vm1, hostNetwork=true)
  → cert-manager TLS termination
  → Studio/Grafana/ArgoCD Service (ClusterIP)
  → 해당 Pod

[아웃바운드]
vm2/3/4/5 (Private)
  → vm1 NAT/MASQUERADE
  → 인터넷
```

### vm1 NAT 설정

- `net.ipv4.ip_forward = 1`
- iptables MASQUERADE: `10.1.0.0/16 → enp3s0`
- KakaoCloud 콘솔: Source/Dest Check 비활성화, Private 라우팅 default → vm1

### TLS-SAN (K3s API)

- `--tls-san 210.109.83.10`
- `--tls-san 10.1.1.10`

## 3. 도메인 + HTTPS

### 도메인

**현재: `peakly.art`** (Gabia 등록 → Route53 위임, 2026-06 전환)

- apex `peakly.art` (+ `www.peakly.art`) → 4K Cinema FE
- 서브도메인 A 레코드 → `210.109.83.10`:
  - `data.peakly.art`
  - `ai.peakly.art`
  - `grafana.peakly.art`
  - `argocd.peakly.art`

> **마이그레이션 이력**: 초기에는 `4kakao.kro.kr`(내도메인.한국 무료 DNS)을 사용했고,
> 서브도메인은 `data/ai/grafana/argocd/cinema.4kakao.kro.kr` 였다.
> Burst/DR 전략(Route53 failover) 도입을 위해 정식 도메인 `peakly.art`를 가비아에서 구매,
> 네임서버를 Route53로 위임했다. 메인 서비스는 `cinema.` 서브도메인 → apex로 변경.
> 전환기에는 양 도메인을 ingress에 함께 두어(dual-host) 무중단 처리.

### 인증서 (cert-manager + Let's Encrypt)

- **현재 상태**: Production (`letsencrypt-prod`) — 모든 Ingress 전환 완료
- HTTP-01 challenge로 자동 발급
- FE Ingress에 `ssl-redirect: true` 적용 → HTTP 접속 시 HTTPS 자동 리다이렉트
- 초기 Staging 사용 이유: kro.kr 도메인이 Public Suffix List 미등록 → 모든 kro.kr 사용자가 rate limit(50개/주)를 공유. 한도 초과 후 다음 주기에 prod 전환 완료

## 6. GPU Operator

### 구성

- NVIDIA Driver 580.126.20
- CUDA Runtime 13.0
- Tesla T4 인식 (`nvidia.com/gpu: 1`)
- vm5에만 배포 (`workload=gpu`, `gpu=nvidia` 라벨)

### containerd 통합 방식

- k3s-agent는 부팅 시 `/usr/local/nvidia/toolkit/nvidia-container-runtime` 바이너리를 감지하면
  **자체 containerd 설정(`/var/lib/rancher/k3s/agent/etc/containerd/config.toml`)에 nvidia 런타임을 자동 등록**한다.
- 따라서 containerd 설정 자체는 **k3s가 관리**하며, GPU Operator toolkit의 역할은 런타임 바이너리·드라이버 설치까지로 한정한다.
- GPU 사용 파드는 `resources.limits`에 `nvidia.com/gpu: 1`을 요청하면 스케줄된다.

## 7. Longhorn (스토리지)

### 구성

- vm4에만 배포 (`workload=data`)
- `defaultReplicaCount: 1` (단일 노드, 복제 없음)
- `local-path`가 default StorageClass, `longhorn`은 명시 지정

### 주요 트러블슈팅

- **uninstall 실패**: `deleting-confirmation-flag` 필요 → `--no-hooks`로 우회 후 수동 정리
- **engineimage 사라짐**: CRD 강제 삭제 시 vm4 호스트의 engine 바이너리도 같이 사라짐 → 전체 재설치
- **backuptarget "default" 없음**: 강제 정리 후 누락 → 수동 BackupTarget 리소스 생성으로 해결

## 9. Supabase 구성

### 배포 방식

- Helm repo: `https://supabase-community.github.io/supabase-kubernetes` (chart v0.5.6)
- **현재 상태: `helm install`로 직접 배포** (릴리스명 `supabase-ai`/`supabase-data`, 각 ns)
  values 파일은 `Ansible/values/values-ai.yaml`, `Ansible/values/values-data.yaml`
- ⚠️ **ArgoCD 관리 대상이 아님.** `kubectl get applications -n argocd`에는 `4k-be`, `4k-fe`, `infra-manifests`만 존재.
  → Supabase는 GitOps self-heal이 걸리지 않으므로, 구성 변경·복구는 `helm upgrade`로 수동 수행한다.
  (`argocd-app-supabase-*.yaml` 매니페스트는 저장소에 있으나 클러스터에 미등록 — ArgoCD 편입은 향후 작업)

### Ingress 구조 (data ns)

`ingress-studio.yaml`에 두 Ingress가 함께 정의됨:

| Ingress 이름 | 경로 | 백엔드 서비스 | 인증 |
|---|---|---|---|
| `supabase-data-studio` | `/*` | Studio:3000 | Basic Auth |
| `supabase-data-api` | `/rest/v1`, `/auth/v1`, `/storage/v1` | Kong:8000 | Supabase JWT (`apikey` 헤더) |

> Studio Ingress가 `/*`를 점유하므로, REST API는 더 구체적인 경로로 **별도 Ingress**가 반드시 필요.  
> Kong을 통하지 않으면 PostgREST, Auth 등 API가 전혀 동작하지 않음.

### data ns (서비스 DB)

- 활성: db, studio, kong, rest, meta, functions
- 비활성: auth, realtime, storage, imgproxy, analytics, vector, minio
- pgvector extension 활성화 (벡터 검색용)
- 메타데이터 + 도파민 시계열 + 임베딩 모두 PG에 저장
- 검색: pgvector cosine + DTW 재순위
- **movies 테이블**: `public` 스키마 (PostgREST 기본 탐색 스키마), RLS 비활성화  
  → 자가호스팅 Supabase에서 `service_role` JWT가 RLS를 자동 우회하지 않으므로 수동 비활성화 필요

### ai ns (학습 DB)

- 활성: db, studio, kong, rest, meta, functions, storage, imgproxy, minio
- Storage는 체크포인트 저장용 (MinIO 백엔드)
- pgvector extension 활성화
- 비활성: auth, realtime, analytics, vector

### Secret 관리

- `Ansible/generate-apply-secrets/generate-secrets.sh` 스크립트로 일괄 생성
- `secrets-data/.env`, `secrets-ai/.env`에 평문 보관 (gitignore)
- 생성되는 secret:
  - `supabase-jwt`, `supabase-db`, `supabase-dashboard`, `supabase-smtp`, `supabase-analytics`, `supabase-s3` (ai만), `supabase-minio` (ai만)

## 12. ArgoCD (GitOps)

### 구성

- vm1에 배포
- 외부 접속: `https://argocd.peakly.art`
- 기본 user: `admin`, 비번: `argocd-initial-admin-secret`

### Applications (현재 클러스터에 등록된 것 = 3개)

`kubectl get applications -n argocd` 기준 실제 등록 현황:

| 이름 | 경로/차트 | 대상 namespace | 비고 |
|---|---|---|---|
| `infra-manifests` | GitHub `Ansible/manifests/apps/` | default | Ingress, ClusterIssuer, ConfigMap |
| `4k-fe` | GitHub `Ansible/manifests/4k-fe/` (Kustomize) | fe | FE Next.js, GitHub Actions → GHCR 이미지 자동 배포 |
| `4k-be` | GitHub `Ansible/manifests/4k-be/` (Kustomize) | be | BE FastAPI, GitHub Actions → GHCR 이미지 자동 배포. 외부 노출 없음(ClusterIP) |

> 각 Application은 `Ansible/manifests/argocd/argocd-app-*.yaml`을 최초 1회 `kubectl apply`로 등록.

> ⚠️ **Supabase(`ai`/`data`)는 ArgoCD에 등록돼 있지 않다.** `helm install`로 직접 배포된 상태(9번 참조)이며,
> GitOps self-heal 대상이 아니다. 구성 복구는 `helm upgrade`로 수동 수행한다.

### 동작 검증

- Git push → 3분 이내 자동 sync
- 수동 변경 시 ArgoCD가 Git 상태로 자동 복원 (selfHeal)

## 16. CI/CD 파이프라인

### 4K Cinema FE

```
git push (4K_FE/** 변경)
  → GitHub Actions (.github/workflows/deploy-4k-fe.yml)
      ① Docker 빌드 (Next.js standalone, multi-stage)
      ② ghcr.io/sanggyoon/4k-fe:<sha> GHCR 푸시
```

## 18. 트러블슈팅 체크리스트

| 증상                     | 확인사항                                             |
| ------------------------ | ---------------------------------------------------- |
| `kubectl` 연결 실패      | KUBECONFIG 환경변수, TLS-SAN, SG 6443                |
| Private VM 인터넷 안 됨  | vm1 NAT 규칙, ip_forward, Source/Dest Check          |
| Pod이 노드에 안 떠짐     | nodeSelector 라벨, taint/toleration                  |
| GPU 인식 안 됨           | nvidia-device-plugin 재시작, containerd config       |
| Longhorn 설치 실패       | open-iscsi, iscsid, CRD/webhook 잔여물, BackupTarget |
| kubelet 502              | 노드 SG 10250, kubelet 동작 여부                     |
| Supabase Pod ConfigError | Secret 키 이름 mismatch (특히 minio, s3)             |
| 인증서 발급 실패         | DNS 전파, rate limit, 80 포트 접근                   |
| Ingress 충돌             | 같은 host 중복 사용                                  |
| ArgoCD sync 안 됨        | Git push 여부, repoURL/path 정확성                   |
| REST API `/rest/v1` 응답 없음 | Studio Ingress가 `/*` 전체를 점유 → Kong용 `supabase-data-api` Ingress 별도 추가 필요 |
| INSERT 42501 (RLS 정책 위반) | 자가호스팅 Supabase에서 `service_role` JWT가 RLS 자동 우회 안 됨 → Studio SQL Editor에서 `ALTER TABLE <table> DISABLE ROW LEVEL SECURITY` 실행 |
